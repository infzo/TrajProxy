# 变更记录技能 (Record Change)

> 分析 git 变更历史，将变更分类后追加到 `docs/release.md`。

---

## 命令格式

| 命令 | 说明 |
|------|------|
| `/record-change` | 自动分析自上次记录以来的变更，追加新条目 |
| `/record-change v1.2.0` | 指定版本号，分析并追加新条目 |
| `/record-change v1.2.0 2026-04-01` | 指定版本号和日期 |

---

## 执行流程

```
用户调用 /record-change [版本号]
        │
        ▼
┌─────────────────────────────────┐
│  步骤1: 读取现有变更日志          │
│  - 读取 docs/release.md         │
│  - 确认最后记录的版本/日期        │
│  - 如果文件不存在,提示先初始化     │
└─────────────┬───────────────────┘
              │
              ▼
┌─────────────────────────────────┐
│  步骤2: 获取 git 变更            │
│  - git log 获取自上次记录以来     │
│    的所有 commit                 │
│  - 如果无版本参数,自动推断版本号  │
└─────────────┬───────────────────┘
              │
              ▼
┌─────────────────────────────────┐
│  步骤3: 分析每个 commit          │
│  - git diff --stat 查看影响范围  │
│  - 对 "xxx"/"xx" 类 commit,     │
│    通过文件变更推断实际内容       │
└─────────────┬───────────────────┘
              │
              ▼
┌─────────────────────────────────┐
│  步骤4: 分类变更                 │
│  根据提交信息和文件变更分类       │
└─────────────┬───────────────────┘
              │
              ▼
┌─────────────────────────────────┐
│  步骤5: 生成条目并追加           │
│  - 按格式生成 markdown 条目      │
│  - 追加到 docs/release.md 顶部  │
│    (最新版本在前)                │
└─────────────────────────────────┘
```

---

## 变更分类规则

分析 commit 信息和文件变更进行分类：

| 分类关键词 | 类型 |
|-----------|------|
| 添加、新增、实现、支持 | 新功能 |
| 优化、重构、整理、调整、改进 | 改进 |
| 修复、fix | Bug 修复 |
| 更新文档、更新镜像 | 改进 |
| 修改、对齐 | 视具体内容而定 |
| "xxx"/"xx"（无意义信息） | 必须通过 git diff --stat 推断实际变更 |

**推断规则**（当 commit 信息无意义时）：
- 新增文件 → 新功能
- 仅修改已有文件少量行 → Bug 修复或改进
- 大量文件变更 → 根据涉及模块判断（架构 → 改进，测试 → 改进，核心模块 → 新功能）

## 条目格式模板

```markdown
## v[版本号] ([日期])

**类型**: [初始版本 / 功能迭代 / 架构重构 / Bug 修复 / 维护更新]

### 新功能
- [功能描述]

### 改进
- [改进描述]

### Bug 修复
- [修复描述]

### 影响范围
- `模块/文件路径`
```

**格式注意**：
- 如果某个分类下没有条目，省略该分类
- 最新版本条目插入到文件顶部（文档头之后、上一版本之前）
- 版本类型判断：单个小修复 → Bug 修复；少量功能 → 功能迭代；涉及多模块重构 → 架构重构；仅文档/配置 → 维护更新

---

## 版本号规则

- 如果用户提供了版本号参数，使用用户指定的版本号
- 如果没有提供版本号：
  - 读取 `docs/release.md` 中最后记录的版本号
  - 自动递增补丁版本号（例如 v0.5.0 → v0.6.0）
  - 如果是大版本变更（架构重构、重大新功能），建议递增次版本号

---

## 边界情况处理

1. **docs/release.md 不存在**：提示用户需要先创建初始变更日志
2. **没有新的 commit**：提示 "自上次记录以来没有新的变更"
3. **commit 信息为 "xxx"**：必须通过 git diff 和 git log --stat 分析实际代码变更来推断内容，不直接使用无意义的提交信息
4. **单次 commit 涉及多个模块**：在影响范围中全部列出
5. **用户指定已存在的版本号**：提示版本号已存在，询问是否要更新该条目

```
# 自动记录最新变更
```
